# Phase 3 — UI Design Contract: Routine Editor

> Visual and interaction contract for the `/editor` route. Generated by gsd-ui-researcher. Derived from REQUIREMENTS.md EDIT-01..EDIT-05, ROADMAP.md Phase 3 success criteria, existing dashboard design system, and research/ARCHITECTURE.md Layer 5 (server-rendered forms, textarea over Monaco).

## Design System

| Property | Value |
|----------|-------|
| Tool | none (project uses bespoke Tailwind 3.4 theme — the "lunar/celestial palette") |

## Typography

4 distinct font sizes (11, 12, 14, 48 px) and 4 inherited weights (400 body / 500 label / 600 heading / Fraunces variable `wght` axis at 320 display + 600 display-bold). No new sizes or weights are introduced this phase — every value below is already shipped in v0.1 via `globals.css` primitives. Fraunces uses variable `wght` axis; Bricolage body uses 400 + 500 only in the Editor.

| Role | Size (px / Tailwind) | Weight | Line Height | Font | Where |
|------|----------------------|--------|-------------|------|-------|
| Display (page title) | 48 / `text-5xl` | Fraunces `wght 320`, `opsz 144`, `SOFT 80` | 0.95 / `leading-[0.95]` | Fraunces | `<PageHeader title="Author a routine">` — one per page |
| Heading (section h2) | 14 / `text-sm` | `font-semibold` (600) | 1.4 | Bricolage | Section titles ("Runtime", "Schedule", "Prompt", "Safety") — reuses `settings-client.tsx` pattern |
| Body | 14 / `text-sm` | 400 | 1.5 / `leading-normal` | Bricolage | Input values, field descriptions |
| Label / eyebrow | 11 / `text-[11px]` | 500 via `font-medium` | 1.4 | JetBrains Mono uppercase tracked (the `.label` utility class, `tracking-[0.18em]`) | Every field label ("Name", "Slug", "Runtime", "Cron") and the page eyebrow ("AUTHORING") |
| Data (slug, cron, paths) | 12 / `text-xs` | 400 | 1.4 | JetBrains Mono (`.data` or `font-mono`) | Slug preview, plist path preview, bundle path preview, cronstrue output |
| Inline validation | 12 / `text-xs` | 400 | 1.4 | Bricolage | Error + warning messages under fields |

**Rules:**
- Headings use `font-semibold` (600); body uses 400. No mid-weights (450/500) in body copy. Exception: `label` utility class uses `font-medium` (500) because it's already established in globals.css.
- Display text uses Fraunces only for the single page title — never for section headings (too much characterful weight competes with data).
- All machine-readable values (slugs, paths, cron expressions, labels, timestamps, plist paths) use `font-mono` / JetBrains Mono.
- `letter-spacing: -0.025em` (`tracking-tighter`) on Fraunces display text only.

## Layout Contract

The research/ARCHITECTURE.md Layer 5 locks "server-rendered form + Server Action." This section is the prescriptive visual layout.

### Grid

Within the parent `<main className="flex-1 px-12 py-10 max-w-6xl">`:

```
┌──────────────────────────────────────────────────────────┐
│ PageHeader (eyebrow / title / subtitle)                  │  mb-10
├──────────────────────────────────────────────────────────┤
│ Draft-recovery banner (conditional, .panel-raised)       │  mb-6
├─────────────────────────────────────┬────────────────────┤
│                                     │                    │
│ Form column (flex-1, max-w-[640px]) │ Preview column     │  gap-8
│                                     │ (w-80, sticky top) │
│   • Name + Slug row                 │                    │
│   • Runtime radio-card grid (2×2)   │ • Bundle path      │
│   • Prompt textarea                 │ • Plist path       │
│   • Schedule + cron preview         │ • Marker tag       │
│   • Reversibility + Budget row      │ • Cronstrue line   │
│   • Save action bar                 │ • Health status    │
│                                     │                    │
└─────────────────────────────────────┴────────────────────┘
```

**Single-column vs two-column decision:** Two-column at ≥ 1024px viewport width (the dashboard's target). At < 1024px the preview column collapses under the form. Preview column uses `sticky top-10` so it stays visible while the user scrolls through the long prompt textarea.

**Rationale:** The form is long (prompt textarea is 30 rows); a scroll-visible live preview of the disk path, plist label, and cronstrue output gives the user immediate "this is what will happen" feedback — the core anti-pitfall for Pitfall #5 (partial-success deploys) starts by making the artifact path legible before the user clicks Save.

### Form column structure

Each field group is a `<section>` wrapped in the existing `Section({title, desc, children})` helper from `settings-client.tsx` (duplicate that helper verbatim into the Editor client, do not extract yet — Phase 4 or 6 can extract it if multiple pages converge). This buys free consistency with Settings.

Field group ordering (top to bottom, user-centric "what, where, when, how" flow):
1. **Identity** — Name + Slug (side-by-side row on ≥ 640px, stacked under)
2. **Runtime** — 2×2 radio-card grid (claude-routines, claude-desktop, codex, gemini)
3. **Prompt** — single textarea, 30 rows, `font-mono`, full-width
4. **Schedule** — cron input + cronstrue preview (aurora pill) directly below
5. **Safety** — Reversibility radio row (green/yellow/red) + Budget number input side-by-side
6. **Action bar** — left-aligned `Save routine` primary + right-aligned `Discard draft` ghost, sticky at bottom when scrolled off (background `bg-ink-900/90 backdrop-blur` mirroring sidebar hairline pattern)

### Runtime radio-card grid (specific component contract)

Each card:
- `<label>` wrapping a hidden `<input type="radio" name="runtime">`
- `.panel` default, `.panel-raised` + `ring-1 ring-dawn-400` on selected, `opacity-40 cursor-not-allowed` on unavailable
- Layout: icon (16px, lucide) + card title (14px font-medium) on row 1; one-line description (12px moon-400) on row 2; health pill (pill-green / pill-amber / pill-muted) on row 3
- Touch target: ≥ 88px × ≥ 80px (two cards per row at 640px form column width = ~300px each)

### Prompt textarea (specific)

- `rows={30}`, `className="w-full bg-ink-900 border border-ink-600 rounded-md px-4 py-3 text-sm font-mono resize-y"`
- MUST include attributes per EDIT-05: `autocomplete="off"` `autocorrect="off"` `autocapitalize="off"` `spellcheck="false"` `data-1p-ignore` `data-lpignore="true"` `data-form-type="other"` `data-bwignore`
- Focus ring inherited from global `:focus-visible` (dawn-400 outline).
- Character count display bottom-right of textarea: `{count} / 16,000` — moon-600 at <80%, signal-amber at 80–99%, signal-red at ≥ 100%
- No syntax highlighting, no autocomplete, no tab-capture. It's a `<textarea>`. Anti-Pattern 6 in research/ARCHITECTURE.md is binding.

### Secret-scan error panel (specific, high-visibility)

When secret scan trips, render directly below the textarea (not as toast, not as modal):

- `.panel` with `border-signal-red/50 bg-signal-red/5`
- Heading: `Secret detected — save blocked` (`text-signal-red` semibold)
- Body: `{matched_pattern_name} at line {n}, column {c}. Replace the matched substring with ${VAR} and document the variable in AUTHORING.md.`
- Inline code example of the fix (mono, panel-raised):
  ```
  Before:  OPENAI_API_KEY=sk_live_abc123…
  After:   OPENAI_API_KEY=${OPENAI_API_KEY}
  ```
- Link: `View matched region` → scrolls textarea to `line:col` and briefly flashes a signal-red background on that line (framer-motion 300ms)

This panel MUST remain visible until the matched substring is removed. It MUST NOT auto-dismiss. The Save button MUST stay in its enabled-but-blocked state (click scrolls to panel, does not submit).

### Draft-recovery banner (specific)

When localStorage has a draft for a different browser session (detected via a session-nonce), render above the form:

- `.panel-raised` with `border-aurora-400/30`
- Left: moon/sleep glyph (reuse `MoonGlyph`) at 20px
- Center: `You have an unsaved draft from {relative time, e.g. "14 minutes ago"}.` + second line (moon-400): `{name || "Unnamed routine"} · {runtime || "no runtime"} · {slug || "no slug"}`
- Right: `Restore draft` (aurora-outlined ghost button) + `Start fresh` (btn-ghost)
- Dismisses on either action; does not reappear on reload once dismissed

**Does NOT silently restore.** Silent restore was considered and rejected — if the user returned expecting a clean slate (e.g. they completed a different routine on another browser tab), silent restore would be confusing. Explicit banner + choice matches the "no surprises" OSS ethos.

## Interaction Contracts

### Autosave (EDIT-03)

- Debounce: **500ms** after last keystroke in any field (matches REQUIREMENTS.md).
- Storage key: `sleepwalker.draft.v1` (namespaced for future schema evolution).
- Payload shape:
  ```json
  {
    "version": 1,
    "sessionNonce": "<random-on-page-load>",
    "updatedAt": "<iso8601>",
    "fields": { "name": "", "slug": "", "runtime": "", "prompt": "", "schedule": "", "reversibility": "", "budget": 0 }
  }
  ```
- Visual signal: After first autosave, show small muted pill in the action bar: `draft saved {relativeTime}` (moon-600, 11px, data font). Updates in place; no toast.
- Clear on successful save: `localStorage.removeItem("sleepwalker.draft.v1")` after Server Action returns `ok: true`.
- In-app navigation intercept: wrap sidebar `<Link>` clicks (only on `/editor` page) with a handler that checks dirtiness and invokes the confirm modal before `router.push`.
- Browser close / refresh: `beforeunload` event registered only while dirty.

### Slug auto-derive (EDIT-04)

- Name field has `onChange`: if slug has never been manually edited, derive slug from name: lowercase, strip non-alphanumeric except `-`, collapse multi-hyphens, trim leading/trailing hyphens, truncate to 64 chars.
- Slug field sets a `touched` flag on user edit; once touched, name changes no longer derive.
- A `↺ Re-derive from name` micro-link (text-xs, aurora-400) appears next to the slug label when `touched === true`, letting the user opt back in.

### Cronstrue live preview

- Import `cronstrue` (already in STACK.md as a net-new dep per CLAUDE.md).
- On every cron-field `onChange`, attempt `cronstrue.toString(value, { verbose: false, use24HourTimeFormat: true })`.
- Catch throw → render inline `Invalid cron — …` in signal-red.
- Success → render `Runs {string}` in aurora-400 pill below input.
- Sleep-window warning: after a successful parse, compute next fire time (use `cron-parser` or hand-rolled) and compare to settings.sleep_window. If outside, render an additional amber helper line: `Fires at {time} which is outside your sleep window ({start}–{end}).` (Non-blocking.)

### Secret scan (EDIT-02)

- Runs on every prompt `onChange` **after a 250ms debounce** (separate from the 500ms autosave debounce — faster because block-on-save requires fast feedback).
- Pattern source: gitleaks default ruleset ported as JS regex array — see research/STACK.md gitleaks reference. Minimum patterns enforced this phase:
  - Stripe live key: `sk_live_[0-9a-zA-Z]{24,}`
  - GitHub PAT: `ghp_[0-9a-zA-Z]{36,}`
  - AWS access key: `AKIA[0-9A-Z]{16}`
  - Generic high-entropy hex ≥ 40 chars: `\b[0-9a-fA-F]{40,}\b`
  - Slack bot token: `xox[baprs]-[0-9a-zA-Z-]{10,}`
  - Private key header: `-----BEGIN (RSA |EC |DSA |OPENSSH |)PRIVATE KEY-----`
- Client-side scan is a UX preview; **server-side scan in the Server Action is authoritative**. The client scan MUST match the server's pattern list exactly (share a `secrets-patterns.ts` module).
- Block Save on any match: Save button stays interactive (for a11y) but click handler checks `hasSecretMatch` and refuses submit, scrolls to error panel.

### Slug collision check

- On every slug `onChange` after the regex passes, call a read-only Server Action (`checkSlugAvailability({runtime, slug})`) debounced 400ms.
- Server Action reads `routines-{runtime}/{slug}/` existence (`fs.access`). Returns `{available: boolean}`.
- Show inline: `Available — will write to routines-{runtime}/{slug}/` (signal-green, 12px) or `Already exists — choose another slug.` (signal-red). Save button also checks this flag.
- Cross-runtime: the check also flags if ANY `routines-*/` directory has the slug (ADPT-02 namespacing rule). Message differs: `A {other_runtime} routine with slug {slug} exists. Slugs must be unique across runtimes.`

### Health-check integration (dims unavailable runtime radios)

- On `/editor` page mount (Server Component), call `healthCheckAll()` from `dashboard/lib/runtime-adapters/index.ts`.
- Pass `healthStatuses` as prop to the client component.
- Each radio card reads its status: `available: true` → full opacity + health pill-green. `available: false` → opacity 40, `cursor-not-allowed`, `pill-amber` with the reason string as tooltip (`title` attr + visible helper below card).
- Selecting an unavailable runtime is disallowed at the input level (`disabled` on the radio). Save button cross-checks: even if somehow a disabled runtime were selected, validation rejects.

### Keyboard and accessibility

- Tab order: name → slug → runtime radios (left-right, top-bottom within grid) → prompt textarea → cron → reversibility radios → budget → Save → Discard.
- `Cmd/Ctrl+S`: invokes Save (intercepts browser save-page).
- `Esc` inside the prompt textarea: no-op (don't steal focus).
- `Esc` in any confirm modal: focuses `Keep editing`.
- All form controls have associated `<label for>` or wrap `<label>`.
- Runtime radio cards use `role="radio"` + `aria-checked` + arrow-key navigation within the group.
- Focus ring is the global `:focus-visible` dawn-400 outline — do not override.
- Color contrast verified: moon-200 on ink-900 = 13.7:1 (AAA); signal-red on ink-900 = 5.1:1 (AA); dawn-400 on ink-900 = 9.2:1 (AAA). All pass WCAG 2.1 AA.
- `prefers-reduced-motion`: framer-motion stagger animations become opacity-only fades (0.2s). No translate.
